# Module 6 — Returns & Credit Note Automation
**Status:** Spec Approved | **Build Status:** Not Started

## Problem Statement
Retailer returns are a major source of financial disputes.
Retailers claim returns for batches never purchased from this depot,
claim higher quantities than originally purchased, or return
products damaged by their own mishandling.
Credit notes issued manually take 2-3 weeks and are frequently disputed.
Fake returns cost distributors ₹50,000-2,00,000 per year.

## Functional Requirements
FR1: Return linked to original invoice at creation (required, not optional)
FR2: Batch number on return verified against original invoice line item
FR3: Photo of returned goods required (damage proof)
FR4: System flags quantity_returned > quantity_sold from original invoice
FR5: System flags batch_number not present in original invoice
FR6: Credit note auto-generated by DB trigger on return approval
FR7: Credit note auto-applied against retailer outstanding balance
FR8: Audit log: who returned, who approved, photo evidence, credit note number
FR9: Manager receives NOTIFY-tier alert for every return above ₹5,000 value

## API Contracts
POST /returns/create
  Request: {
    original_invoice_id: uuid,
    batch_id: uuid,
    product_id: uuid,
    quantity_returned: int,
    return_reason: string,
    photo_proof_url: string,
    created_by: user_id
  }
  Response: {
    return_id, is_fake_return_flagged: bool,
    flag_reason: string | null,
    status: "PENDING_APPROVAL"
  }
  Side effects: returns record created, AnomalyAgent notified if flagged

POST /returns/{id}/approve
  Request: { decided_by: user_id, notes: string }
  Response: { return_id, credit_note_id, credit_note_amount, status: "APPROVED" }
  Side effects: credit_notes created (via DB trigger), stock_movement (RETURN),
                audit_trail, retailer.current_outstanding reduced

POST /returns/{id}/reject
  Request: { decided_by: user_id, rejection_reason: string }
  Response: { return_id, status: "REJECTED" }
  Side effects: audit_trail, no credit note created

GET /returns/{id}/credit-note
  Response: PDF credit note with return details, amount, batch info,
            original invoice reference, depot stamp

GET /returns/history/{depot_id}
  Query params: retailer_id (optional), from_date, to_date, status
  Response: [{ return_id, retailer, product, qty, amount,
               status, credit_note_id, created_at }]

## Fake Return Detection Rules
Flag 1: quantity_returned > original invoice line item quantity
Flag 2: batch_number not in original_invoice.line_items
Flag 3: return created > 90 days after original invoice date
Flag 4: same retailer, same batch, same product returned twice within 30 days
Any flag → is_fake_return_flagged = TRUE, requires manager APPROVE not auto-approve

## Credit Note Auto-Generation (DB Trigger)
Trigger fires AFTER UPDATE ON returns WHERE is_fake_return_flagged = FALSE
and manager approves (outcome = APPROVED)
INSERT into credit_notes: amount = quantity_returned × ptr from original line item
Status sequence: ISSUED → APPLIED (when offset against next invoice) → SETTLED

## Stock Movement on Return
RETURN movement created only after manager approval
Batch re-enters stock_movements as RETURN type
If batch is expired: RETURN movement created but batch immediately flagged for WRITE_OFF
FEFO ranking updated in next nightly run

## Edge Cases
- Partial return: 10 units of original 50 → credit note for 10 units, 40 remain invoiced
- Return of cold chain batch: verify temperature compliance during transport before accepting
- Return of recalled batch: automatically APPROVED, no fake-return flag possible
- Return of batch under ANOMALY_HOLD: require ADMIN approval, not just MANAGER
- Credit note applied to multiple invoices: split credit across oldest invoices first (FIFO)
- Retailer disputes rejection: escalate to DISPUTE status, excluded from aging until resolved
