--------------------------------------------------------------------------------
  SECTION 1 — WHAT IS FLOWSYNC
--------------------------------------------------------------------------------

FlowSync Health is a B2B SaaS platform for Indian pharmaceutical depots.
It solves five problems that cost each depot ₹6–11 lakhs per year:

  1. Expiry losses         — medicines expire on shelves undetected (₹3–6L/year)
  2. Billing chaos         — manual invoice typing errors, ghost stock (₹1–2L/year)
  3. Cold chain failures   — fridge temperature excursions, fake logs
  4. Payment confusion     — receipts unmatched to invoices for months
  5. Compliance gaps       — GDP, Rule 65, Schedule M audit violations

The system combines an ML prediction layer, a multi-agent decision engine,
a FastAPI backend, IoT temperature sensors, and a mobile-first frontend.

Target market: ~1.5 lakh licensed pharmaceutical depots in India.

--------------------------------------------------------------------------------
  SECTION 2 — SYSTEM ARCHITECTURE OVERVIEW
--------------------------------------------------------------------------------

  INBOUND DATA SOURCES
  ├── Tally / Marg ERP exports (CSV/Excel) — stock movements, invoices
  ├── Field rep mobile app — payment photos, GPS, returns
  ├── IoT sensors — MQTT temperature every 5 minutes
  └── OCR pipeline — photo invoices from WhatsApp / camera

  CORE LAYERS (bottom to top)
  ├── PostgreSQL (multi-tenant, RLS per depot)
  ├── ML Layer (ml/) — nightly predictions + auto-training
  ├── Agent Layer (ml/agents/) — decisions from predictions
  ├── FastAPI Backend (backend/) — REST API
  ├── Celery + Redis — async tasks, nightly pipeline trigger
  └── Frontend (not started)

  NIGHTLY PIPELINE (2 AM, triggered by Celery)
  ┌─────────────────────────────────────────────────────────────────────┐
  │  Stage 1: Demand forecast (XGBoost) → demand_predictions table      │
  │  Stage 2: Expiry risk (RF + Calibrated) ─┐  (parallel asyncio)      │
  │  Stage 2: Stockout risk (formula)        ─┘                         │
  │  Stage 3: FEFO ranking (sort + ML override)                         │
  │  Stage 4: Anomaly detection (Z-score) ← background, non-blocking    │
  │  Stage 5: All 4 agents run on shared state (in-memory, sync)        │
  │  Stage 6: Coordinator resolves conflicts, writes agent_actions       │
  └─────────────────────────────────────────────────────────────────────┘

  Every stage has a fallback to yesterday's data if it fails.
  Every outcome is written to pipeline_run_logs.

  MULTI-TENANCY MODEL
  Every tenant (depot) has RLS enforced at the PostgreSQL level.
  FastAPI middleware reads depot_id from JWT and runs:
    SET LOCAL app.current_depot_id = '{depot_id}'
  before every query. RLS policies on every tenant table enforce this.
  ML pipeline uses a separate admin connection that bypasses RLS.
  The admin connection is NEVER exposed through any HTTP route.

--------------------------------------------------------------------------------
  SECTION 4 — ML LAYER: WHAT IS BUILT AND HOW IT WORKS
--------------------------------------------------------------------------------

The entire ML layer is production-ready. Here is what each file does.

  ml/shared/config_loader.py
    The ONLY place thresholds exist at runtime.
    get_config(db_session) reads compliance_config table at 2 AM.
    Falls back to _defaults() if DB is unreachable.
    Used by every inference file. Cache is cleared at start of each pipeline run.

    Key thresholds and their defaults:
      fefo_override_threshold:     0.60
      expiry_critical_threshold:   0.85  ← SUGGEST_LIQUIDATION fires above this
      expiry_warning_threshold:    0.60  ← FLAG_PRIORITY_DISPATCH fires above this
      anomaly_hold_threshold:      2.5   ← ANOMALY_HOLD fires above this Z-score
      anomaly_alert_threshold:     2.0   ← ANOMALY_ALERT fires above this Z-score
      default_lead_time_days:      7
      cashflow_reorder_reduction:  0.30  ← Reorder qty reduced by 30% in cashflow crunch

  ml/features/
    demand_features.py  — builds XGBoost feature matrix from sales data
                          (lag features, rolling means, festival flags, dow/month)
    expiry_features.py  — builds RF feature matrix from batch data
                          (velocity ratio, time pressure, category risk)
    anomaly_features.py — builds Z-score features from stock movements
                          (category baseline mean/std, lookback window)

  ml/inference/orchestrator.py
    Entry point: run_inference_pipeline(depot_id, run_date, db)
    Entry point: run_all_depots(run_date, db)  ← called by Celery task
    Runs all 6 stages in strict order (see architecture diagram above).
    Never change the stage order.

  ml/inference/infer_demand.py
    Loads XGBoost demand model from model_store.
    Predicts 14-day demand per product.
    Writes to demand_predictions table.
    Cold start: if no client model exists, uses global Kaggle-trained model.

  ml/inference/infer_expiry.py
    Loads RF + CalibratedClassifierCV model.
    Falls back to formula if model not trained yet (< 180 days of data).
    Formula: risk = (days_sold / shelf_life) * velocity_pressure
    Writes to expiry_predictions table.

  ml/inference/infer_stockout.py
    Pure formula (no ML model needed — works day one).
    days_until_stockout = current_stock / daily_demand_rate
    Writes to stockout_predictions table.

  ml/inference/infer_fefo.py
    Sorts all active batches by expiry_date ASC (legally required).
    ML override: if expiry_risk_score > fefo_override_threshold, batch
                 gets priority_score = risk * -10 (pushed to top).
    Writes to fefo_rankings table.

  ml/inference/infer_anomaly.py
    Runs as background task (Stage 4) — does not block agents.
    Z-score per stock movement against product-depot baseline.
    Writes to anomaly_flags table.

  ml/agents/
    All agents are synchronous BaseAgent subclasses.
    They operate purely in-memory on a state dict — no DB calls.
    The orchestrator queries the DB after inference and builds the state dict.

    expiry_agent.py    — reads state["expiry_predictions"]
                         → SUGGEST_LIQUIDATION (APPROVE) if risk > critical
                         → FLAG_PRIORITY_DISPATCH (NOTIFY) if risk > warning

    reorder_agent.py   — reads state["stockout_risks"]
                         Guard 1: suppresses reorder if liquidation active for same product
                         Guard 2: reduces qty 30% if state["cashflow_negative"] is True
                         → SUGGEST_REORDER (APPROVE) if days < lead_time
                         → SUGGEST_REORDER (NOTIFY) if days < lead_time * 1.5

    dispatch_agent.py  — reads state["fefo_rankings"]
                         Skips any batch in state["held_batches"] (ANOMALY_HOLD)
                         → DISPATCH_PLAN (AUTO) for every eligible batch with demand

    anomaly_agent.py   — reads state["anomaly_flags"]
                         → ANOMALY_HOLD (APPROVE) — physically blocks batch
                         → ANOMALY_ALERT (NOTIFY) — manager notified, not blocked

    coordinator.py     — instantiates all 4 agents, runs them, deduplicates, sorts by priority
                         Priority order (NEVER change):
                           1. ANOMALY_HOLD
                           2. SUGGEST_LIQUIDATION
                           3. FLAG_PRIORITY_DISPATCH
                           4. SUGGEST_REORDER
                           5. DISPATCH_PLAN
                           6. ANOMALY_ALERT
                         Conflict rule: suppresses SUGGEST_REORDER when
                           SUGGEST_LIQUIDATION is active for same product.
                         Returns list of dicts written to agent_actions table.

  APPROVAL TIERS (defined in ml/agents/base.py):
    AUTO    — executes without human, no notification
    NOTIFY  — WhatsApp fires immediately, operation proceeds
    APPROVE — operation BLOCKED until manager acts or 24h expires

  ml/pipeline/auto_trainer.py
    Triggered by: new client upload, Sunday midnight Celery task, PSI drift alert.
    Flow: ingest → map schema → validate data sufficiency → fine-tune → A/B test → register.
    Fine-tuning uses XGBoost warm-start from global model weights.
    Only deploys fine-tuned model if it beats global MAPE on client's own data.

    COLD START RULES (from ml/pipeline/data_validator.py):
      < 90 days data          → USE_GLOBAL_MODEL (global Kaggle model deployed)
      90–179 days data        → SUFFICIENT_FOR_DEMAND (fine-tune demand only)
      180+ days + 50 labels   → SUFFICIENT_FOR_BOTH (fine-tune both models)

  ml/registry/model_store.py
    ModelStore.save(model, name, metadata) → S3, with local fallback
    ModelStore.load(name)                  → loads latest active version
    ModelStore.rollback(name)              → promotes previous version
    Local fallback dir: ml/models/saved/

  ml/drift/psi_monitor.py
    PSI thresholds: PSI < 0.10 = stable, PSI > 0.20 = trigger retraining.
    Run after every inference batch to detect feature distribution shift.

--------------------------------------------------------------------------------
  SECTION 7 — CRITICAL RULES (NEVER VIOLATE)
--------------------------------------------------------------------------------

These rules are in .claude/rules/. Read those files too. Violations here
will cause legal issues, production bugs, or data loss.

  DATABASE:
    1. Never add UPDATE or DELETE permissions to audit_trail or temperature_logs.
       This is an IT Act 2000 and WHO GDP compliance requirement.
    2. Every tenant-scoped table must have RLS enabled.
    3. Never expose the admin_engine or admin_db_url through any HTTP route.
    4. The Celery Beat scheduler must run as exactly ONE replica. Never scale it.

  ML MODELS:
    5. Never remove class_weight='balanced' from the expiry risk model.
    6. Never remove CalibratedClassifierCV from the expiry risk model.
       Both are required for calibrated probabilities at the decision threshold.
    7. Never change the COORDINATOR_PRIORITY order in ml/agents/base.py.
    8. Never change the stage order in ml/inference/orchestrator.py.
    9. Never duplicate feature engineering code between training and inference.
       Training uses ml/features/, inference uses the same ml/features/ files.

  CONFIG:
    10. Never hardcode these threshold values anywhere except compliance_config seed:
        0.85, 0.60, 2.5, 0.6 (and their variants)
        Always read from config_loader.get() or config_loader.get_config().

  RECONCILIATION (Module 5):
    11. The 4-rule matching engine uses first-match-wins. Rule order is fixed:
        Rule 1: Exact match (±1%, same retailer) → SETTLED
        Rule 2: Partial (< invoice, same retailer, ≥10%) → PARTIAL
        Rule 3: Consolidated (≈ sum of 2-5 invoices ±2%) → SETTLED all
        Rule 4: Advance (no invoice, retailer known) → ADVANCE
        Fallback: UNLINKED → manual review

  CELERY:
    12. celery_beat runs as ONE replica only.
    13. celery_worker scales horizontally, max_concurrency=4 per worker.

  FEFO:
    14. Default sort is expiry_date ASC. This is legally required under
        Schedule M of the Drugs & Cosmetics Rules.
    15. ML override (risk > fefo_override_threshold) only activates when
        BOTH conditions are true: risk > threshold AND batch has a model score.

  PRICING:
    16. Never change pricing without founder (Arghyadeep) sign-off.

  SECTION 10 — HOW TO RUN LOCALLY
--------------------------------------------------------------------------------

  Prerequisites:
    Python 3.11+
    PostgreSQL 14+
    Redis
    (Optional) AWS credentials for S3 (model store falls back to local if absent)

  Setup:
    python -m venv .venv
    .venv\Scripts\activate       (Windows)
    source .venv/bin/activate    (Linux/Mac)
    pip install -r requirements.txt

  Start API server:
    uvicorn backend.main:app --host 0.0.0.0 --port 8000 --reload

  Start Celery worker:
    celery -A backend.tasks.celery_app worker --concurrency=4 --loglevel=info

  Start Celery beat (EXACTLY ONE replica — never more):
    celery -A backend.tasks.celery_app beat --loglevel=info

  Run ML pipeline manually for one depot (dev testing):
    python -c "
    import asyncio
    from backend.database import get_admin_db
    from ml.inference.orchestrator import run_inference_pipeline
    from datetime import date

    async def main():
        async with get_admin_db() as db:
            status = await run_inference_pipeline(
                depot_id='YOUR-DEPOT-UUID',
                run_date=date.today().isoformat(),
                db=db
            )
            print(status)

    asyncio.run(main())
    "

  Run auto-trainer on a CSV file (client onboarding):
    python -c "
    from ml.pipeline.auto_trainer import AutoTrainer
    trainer = AutoTrainer()
    result = trainer.run(
        client_id='test-client-001',
        file_path='data/rossmann/train.csv',
        trigger='onboard'
    )
    print(result)
    "

  API docs (dev only, disabled in production):
    http://localhost:8000/docs

  SECTION 11 — KNOWN ISSUES AND TECH DEBT
--------------------------------------------------------------------------------

  MINOR — config not from compliance_config table in two places:
    ml/features/expiry_features.py, lines 156-167
      CATEGORY_RISK dict is hardcoded.
      Should be moved to compliance_config seed and read via config_loader.

    ml/inference/infer_expiry.py, line 140
      Expiry threshold loaded from model_store metadata instead of config_loader.
      Should read from config_loader.get("expiry_critical_threshold").

  NOT A BUG — intentional design:
    ml/inference/_load_yesterday uses f-string for table name.
    This is safe — table name only comes from 3 hardcoded callsites
    within the same function. Not user-controlled.

    backend/database.py get_db() uses f-string for depot_id in SET LOCAL.
    depot_id is extracted from a verified JWT by auth_middleware before
    reaching get_db(). Auth must validate depot_id is a valid UUID first.

  TRAINING DATA NOTE:
    All models were trained on public Kaggle datasets, not real client data.
      - DemandForecaster: trained on Rossmann + M5 + OTC pharma sales
      - ExpiryRiskModel: trained on pharma_sales_data (proxy labels)
    Real client fine-tuning happens via auto_trainer.py once clients are
    onboarded with 90+ days of data. Global models serve until then.
